```yaml
# .github/workflows/build-and-push.yml
#
# Phase 25 / v3.5-P1: Builds the hudsonfam Dockerfile and pushes to GHCR
# with YYYYMMDDHHmmss UTC timestamp + latest tags. Consumed by Flux
# imagepolicy (Phase 26) via numerical-ordering regex ^\d{14}$.
#
# Action versions pinned to v5/v3 majors per 25-CONTEXT.md D-05/06/07.
# A Node 24 migration (v7/v4 majors) is deferred past v3.5-P4; the
# GitHub Actions runner forces Node 24 default on 2026-06-02 and
# removes Node 20 on 2026-09-16. Current versions remain functional
# inside that window.

name: build-and-push

on:
  push:
    branches: [main]
  workflow_dispatch:

concurrency:
  group: build-and-push-main
  cancel-in-progress: true

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Generate timestamp tag
        id: ts
        run: |
          timestamp="$(date -u +%Y%m%d%H%M%S)"
          echo "ts=${timestamp}" >> "$GITHUB_OUTPUT"
          echo "Timestamp tag: ${timestamp}"

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./Dockerfile
          platforms: linux/amd64
          push: true
          tags: |
            ghcr.io/hudsor01/hudsonfam:${{ steps.ts.outputs.ts }}
            ghcr.io/hudsor01/hudsonfam:latest
          cache-from: type=gha,scope=build-and-push
          cache-to: type=gha,scope=build-and-push,mode=max
```

Decision traceability:
- D-01: single job `build-and-push`, `ubuntu-latest`, `contents: read` + `packages: write`
- D-02: `push.branches: [main]` + `workflow_dispatch:` (no PR trigger)
- D-03: timestamp step `id: ts` with `date -u +%Y%m%d%H%M%S`, stored via `$GITHUB_OUTPUT`
- D-04: two-tag list (`steps.ts.outputs.ts` + `latest`) in `tags:` input
- D-05: `docker/build-push-action@v5`, `platforms: linux/amd64`, `push: true`, `cache-from/to: type=gha,scope=build-and-push`; `mode=max` added per research recommendation (caches all 3 Dockerfile stages, not just final — within D-05 discretion)
- D-06: `docker/setup-buildx-action@v3`, no `with:` block (docker-container driver default, no `--load`)
- D-07: `docker/login-action@v3`, `ghcr.io`, `github.actor`, `secrets.GITHUB_TOKEN`
- D-08: `timeout-minutes: 20` defensive cap (double the 10-min CICD-03 SC)
- D-09: Dockerfile NOT referenced for modification — only invoked via `file: ./Dockerfile`
- D-10: `concurrency.group: build-and-push-main`, `cancel-in-progress: true`
- D-11: no `continue-on-error` or retry on build-push step
- D-12: only `secrets.GITHUB_TOKEN` used; no custom repo secrets

CRITICAL: Do NOT add `docker/metadata-action`, cosign, SBOM steps, matrix strategy, or `sbom: true` — all deferred per 25-CONTEXT.md Deferred Ideas.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub push event → Actions runner | Any commit to main or manual dispatch triggers a build; only repo collaborators with write access can push to main |
| Actions runner → GHCR | Runner authenticates with ephemeral GITHUB_TOKEN scoped to `packages: write` for this repo only |
| Repo write access → workflow file | Anyone with write access to the repo can modify .github/workflows/; mitigated by sole-owner model |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Spoofing | GITHUB_TOKEN authentication to GHCR | accept | GITHUB_TOKEN is ephemeral (expires at run end), auto-rotated, scoped to single workflow run; no long-lived credential exposed |
| T-25-02 | Tampering | `.github/workflows/build-and-push.yml` — malicious workflow modification | accept | Sole-owner repo; git history shows all changes; branch protection (force-push disabled) is standard GitHub default; formal branch protection rules are post-v3.5 backlog |
| T-25-03 | Tampering | Action supply-chain (`docker/build-push-action@v5` etc.) | mitigate | Pinned to major version (`@v5`, `@v3`) — security fixes within the major flow in automatically; full SHA pinning deferred; Renovate/Dependabot monitoring for major bumps is post-v3.5 |
| T-25-04 | Tampering | GHA cache poisoning via forked PR | accept | D-02 excludes `pull_request` trigger entirely; forks cannot write to upstream `type=gha` cache scope; implicit protection from `type=gha` semantics |
| T-25-05 | Information Disclosure | `secrets.GITHUB_TOKEN` in logs | accept | GitHub Actions automatically redacts all `secrets.*` values from log output; canonical YAML has no `echo $GITHUB_TOKEN` in any `run:` step |
| T-25-06 | Denial of Service | Runaway build consuming GitHub-hosted runner minutes | mitigate | `timeout-minutes: 20` job-level cap (D-08 / Claude's Discretion) kills stuck builds; `concurrency: cancel-in-progress: true` (D-10) prevents queue pile-up |
| T-25-07 | Elevation of Privilege | Overly broad job permissions granting write access to repo contents | mitigate | `permissions: contents: read, packages: write` — minimum scope per D-01; no `id-token: write`, no `contents: write`; principle of least privilege enforced |
| T-25-08 | Elevation of Privilege | Fork-triggered workflow with elevated permissions (pwn requests) | accept | D-02 disallows `pull_request` trigger; workflow only runs on `push.branches: [main]` and `workflow_dispatch` — both require repo write access to trigger with write-level permissions |
</threat_model>
