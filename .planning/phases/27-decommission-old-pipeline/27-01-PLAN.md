## PAT-Handling Discipline (T-27-02 mitigation — BLOCKING)

When prompting owner for `WOODPECKER_PAT` (Task 27-01-02) or `FORGEJO_PAT` (Task 27-01-05):

1. **Use a hidden-input prompt mechanism.** Prefer `read -rs PAT_VAR` (bash silent read) inside a heredoc bash invocation, OR use AskUserQuestion's "Other" free-text field if the executor harness provides it. Do NOT ask owner to paste the PAT into a normal chat message that gets transcribed.

2. **NEVER echo the PAT in any tool call argument that gets logged.** This means:
   - Do NOT pass the PAT as an inline argument to a Bash command (e.g., `curl -H "Authorization: token ghp_xxx"` is FORBIDDEN — the entire command is logged)
   - Instead: read PAT into a shell variable inside a heredoc, use `"$FORGEJO_PAT"` interpolation in the curl command, then `unset FORGEJO_PAT` immediately after the loop completes
   - The Bash tool description (passed to the parent agent) should say "Run Forgejo per-version DELETE loop using owner-supplied PAT in shell variable" — NOT include the literal PAT value

3. **NEVER include the PAT in commit messages, file contents, or SUMMARY.md.** SUMMARY.md should reference the PAT only by name (e.g., "FORGEJO_PAT supplied inline") — never the literal value.

4. **UNSET the variable immediately after use.** After the DELETE loop completes (success or failure), run `unset FORGEJO_PAT` (and `unset WOODPECKER_PAT` for Task 27-01-02). This prevents PAT leakage to any subsequent shell command in the same session.

5. **Advise owner to ROTATE the PAT post-execution.** Both PATs are high-scope (Forgejo `write:package` = destructive package operations; Woodpecker Bearer = full repo-management write). Even if no leakage occurred, the SUMMARY.md should note: "Owner: rotate FORGEJO_PAT and WOODPECKER_PAT at <Forgejo UI URL> and <Woodpecker UI URL> within 24h of Phase 27 close as defense-in-depth per T-27-02."

Do NOT use `set -e` semantics inside the loop body — accumulate failures and report at the end.
