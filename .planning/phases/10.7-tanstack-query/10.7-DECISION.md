# 10.7 TanStack Query — Decision: Deferred

**Status**: Not installing
**Date**: 2026-03-28
**Evaluation**: Plan 10.7-01

## Architecture Audit

### Data Fetching (reads)

Every dashboard page is an async server component that queries Prisma directly:

- `dashboard/page.tsx` — `Promise.all()` of 6 Prisma counts + 2 `findMany` calls
- `dashboard/posts/page.tsx` — `prisma.blogPost.findMany()`
- `dashboard/events/page.tsx` — Two `prisma.event.findMany()` calls (upcoming + past)
- All other list pages follow the same pattern

There are **zero client-side data fetches** in dashboard list/detail views. The only
`fetch()` call in the dashboard is the photo upload form (`POST /api/photos`), which is
a file upload — not a query.

Pages use `export const dynamic = "force-dynamic"` and re-fetch on every navigation.

### Mutations (writes)

All mutations are Next.js server actions defined in `src/lib/dashboard-actions.ts`
(and `memorial-actions.ts`). Two patterns:

1. **Form submissions** — Server action bound to `<form action={...}>` or called from
   client form component. After mutation: `revalidatePath()` + `redirect()`.

2. **Inline actions** (delete, ban, approve) — Server action closure passed as prop to
   a thin client component. The client component manages its own loading/error state:

```tsx
// Typical pattern in PostActions, EventActions, PhotoActions, etc.
const [loading, setLoading] = useState(false);
async function handleDelete() {
  setLoading(true);
  try {
    await deleteAction();
    toast.success("Post deleted");
  } catch (err) {
    toast.error(err instanceof Error ? err.message : "Failed to delete");
  } finally {
    setLoading(false);
    setConfirmOpen(false);
  }
}
```

This pattern appears in 6 action components and is ~10 lines each.

### Client-side state

- No client-side query cache
- No SWR or polling
- No background refetch
- Cache invalidation handled entirely by `revalidatePath()` after server actions

## Why TanStack Query Would NOT Add Value

| TanStack Query Feature       | Applicability                                      |
|-----------------------------|----------------------------------------------------|
| Client-side query cache     | No client queries exist — all data from server     |
| Background revalidation     | Server components re-fetch on navigation           |
| Query deduplication         | No duplicate client fetches to deduplicate         |
| Optimistic updates          | No query cache to optimistically update            |
| `useMutation` loading state | Already handled by `useState(false)` + try/catch   |
| `useMutation` error retry   | Delete/ban actions should not auto-retry            |
| `useMutation` onSuccess     | Already handled by `toast.success()`               |

### Cost of integration

- ~13KB gzipped added to client bundle
- New `QueryClientProvider` wrapper with no queries to serve
- `useMutation` would replace 5 lines of `useState` with 5 lines of `useMutation` config — net zero improvement
- Additional dependency to maintain and upgrade

## When to Revisit

TanStack Query would become valuable if the app adds:

- **Client-side search/filtering** (e.g., instant search across posts without full page reload)
- **Real-time or polling features** (e.g., live family update feed)
- **Infinite scroll** with client-side pagination (currently server-rendered)
- **Cross-component cache sharing** (e.g., photo picker that shares cache with photo gallery)
- **Optimistic list updates** (e.g., removing a deleted post from the list before server confirms)

None of these patterns exist today.

## Conclusion

The app follows a server-first architecture where TanStack Query has no meaningful role.
Server components handle all reads, server actions handle all writes, and `revalidatePath()`
handles cache invalidation. The existing `useState` + try/catch pattern for mutation
loading states is simple, consistent, and adequate. Installing TanStack Query would add
complexity and bundle size without improving the user experience.
