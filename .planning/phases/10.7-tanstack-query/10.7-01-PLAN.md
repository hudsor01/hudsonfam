<objective>
Evaluate TanStack Query and optionally integrate for mutation state management.

Purpose: The app uses Next.js server components with Prisma for data fetching and server actions for mutations. TanStack Query may add value for optimistic updates, mutation state tracking (loading/error/success), and background revalidation — but it may also be unnecessary overhead given the server component architecture. This phase audits the current patterns and makes an informed decision.
Output: Documented evaluation with TanStack Query integrated if beneficial, or documented reasoning for skipping.
</objective>

<task type="auto">
  <name>Task 1: Audit current data fetching patterns and evaluate TanStack Query benefit</name>
  <files>audit document</files>
  <action>Systematically audit current data fetching and mutation patterns across the app:

  Data Fetching:
  - Catalog all server components that fetch data via Prisma (pages with direct DB queries)
  - Identify any client-side data fetching (useEffect + fetch, SWR, etc.)
  - Note any patterns where data becomes stale after mutations (e.g., list not updating after create/delete)

  Mutations:
  - Catalog all server actions and how they're invoked (form action, useFormState, direct call)
  - Identify mutation UX gaps: missing loading states, no optimistic updates, no error recovery
  - Note where revalidatePath/revalidateTag is used vs where manual refresh is needed

  Evaluate TanStack Query value:
  - Optimistic updates: Would any mutations benefit from instant UI feedback before server confirmation? (e.g., delete removing row immediately, status toggle)
  - Background revalidation: Are there stale data issues that staleTime + refetchOnWindowFocus would solve?
  - Mutation state tracking: Would useMutation's isLoading/isError/isSuccess improve UX over current patterns?
  - Cost: Additional bundle size, provider setup, potential conflict with server component caching

  Document findings and recommendation in a brief audit summary.</action>
  <verify>Audit document is comprehensive and recommendation is clearly justified</verify>
  <done>Complete audit of data fetching/mutation patterns with clear recommendation on TanStack Query adoption</done>
</task>

<task type="conditional">
  <name>Task 2: Integrate TanStack Query if beneficial, or document skip reasoning</name>
  <files>package.json, src/components/providers.tsx, selected action files</files>
  <action>IF Task 1 recommends integration:
  - bun add @tanstack/react-query @tanstack/react-query-devtools
  - Create QueryClientProvider in src/components/providers.tsx alongside existing providers
  - Configure sensible defaults (staleTime, gcTime, refetchOnWindowFocus)
  - Migrate 2-3 key mutation flows to useMutation for demonstration:
    - Delete post/event actions — add optimistic removal from list
    - Form submission — use useMutation for loading/error/success state tracking
    - Status toggle (if exists) — optimistic update with rollback on error
  - Add ReactQueryDevtools in development mode
  - Ensure server component data fetching is NOT disrupted (keep RSC + Prisma as primary read path)

  IF Task 1 recommends skipping:
  - Document the reasoning clearly (e.g., "server actions + revalidatePath handles most cases, optimistic updates not worth the complexity for this app's scale")
  - Identify specific future triggers that would warrant revisiting (e.g., "if real-time features are added" or "if mutation UX complaints arise")
  - Create summary documenting the decision for future reference</action>
