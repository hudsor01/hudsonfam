<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| n8n pipeline → Postgres (n8n DB) | ALTER TABLE changes schema; workflow UPDATEs write base64 blobs. |
| Stirling-PDF service → n8n (HTTP Request response) | PDF bytes cross the cluster network unauthenticated (internal, trusted). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01-01 | Tampering | ALTER TABLE rollback risk | accept | Change is additive (`ADD COLUMN IF NOT EXISTS pdf_data TEXT`) — no data loss path. Rollback is `DROP COLUMN pdf_data` which would destroy any PDFs already persisted but never compromises other columns. |
| T-21-01-02 | Information Disclosure | PDF content leak via DB backups | accept | PDFs are owner-only artifacts; the n8n DB is already backed up with encrypted volumes per the homelab backup policy. No new leakage surface beyond what cover_letters.pdf_data already established. |
| T-21-01-03 | Denial of Service | Stirling-PDF large-input OOM | accept | Owner-triggered only; concurrency is 1 at a time per the Application Packager's single-row SELECT LIMIT 1. Tailored resume markdown is typically < 10 KB; the HTML wrapper adds ~1 KB; Stirling handles this trivially. |
| T-21-01-04 | Tampering (SQL injection via resume content) | Store Resume PDF node | mitigate | The UPDATE uses parameterized bindings (`$1`, `$2`) via n8n's Postgres node binding syntax. Resume content never interpolates into SQL text. Workflow author must verify the bindings panel uses `{{$json.pdf_base64}}` / `{{$json.tailored_resume_id}}` as parameter values, NOT as string-interpolated SQL. |
</threat_model>
