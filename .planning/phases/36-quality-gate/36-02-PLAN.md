The 8 surviving public pages and their data sources:
- `/`                     Home — Prisma (albums, events) + MDX featured recipes. Watch: image load errors, MenuProvider localStorage hydration (SSR-safe guard in place).
- `/recipes`             Recipe index — MDX file-based, static. Expected clean.
- `/recipes/[slug]`      Recipe detail — MDX static. Use any valid slug from content/recipes/ (e.g. `pangolin`). RecipeChecklist uses localStorage (SSR-safe useEffect guard in place).
- `/photos`             Album listing — Prisma (Album). Phase 34 fixed the R2 proxy.
- `/photos/[album]`      Album detail — Prisma (Album + Photo). Use `moving-to-dallas` (the album seeded/assigned in Phase 34).
- `/events`             Events listing — Prisma (Event). Empty-state guard in place.
- `/richard-hudson-sr`   In Memory — Prisma (Memory + MemorialMedia + MemorialContent).
- `/my-menu`            Build-your-own menu — localStorage (`hudson-menu` key), client-hydrated. Test BOTH empty and populated (>=1 recipe added) states.

"Loads without console errors" definition (RESEARCH.md): zero `console.error` and zero React-originated `console.warn` (hydration mismatch, missing key, etc.) in browser DevTools during initial load AND after hydration settles (~2s). Image 404s count. Network errors for required API calls count.

NOT a failure: the Node `ExperimentalWarning: localStorage ...` that appears in `npm test`/Node output — it is a Node runtime warning from happy-dom, never a browser console error (Pitfall 3). Only count entries observed in the browser DevTools console.

Precondition (Pitfall 4): `npm run dev` must be reaching Neon successfully (DB queries return data) before the dynamic-page sweep, so spurious infra errors are not misattributed to code.
