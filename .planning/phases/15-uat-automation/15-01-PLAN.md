<objective>
Run an exhaustive automated browser test against production that exercises every interactive feature of the jobs dashboard. This is not a "does it render" test — it's a "can I break it" test that simulates a real user workflow through every operation.

The test should:
1. Login and navigate to /admin/jobs
2. Verify table view — sort every column, verify order changes
3. Apply each filter type individually and in combination, verify results
4. Search by text, verify matching
5. Switch to kanban view, verify all 6 columns
6. Change a job through every pipeline status (new → interested → applied → interview → offer → rejected), verifying persistence at each step
7. Open detail sheet for multiple jobs, verify content loads
8. Dismiss a job, verify it moves to dismissed tab
9. Restore the dismissed job, verify it returns to active
10. Switch between table/kanban repeatedly to catch rendering bugs
11. Check console for errors after every major operation

Output: Detailed test results with pass/fail per operation and any errors discovered.
</objective>

  <name>Task 2: Exhaustive operations test</name>
  <files>N/A (browser automation only)</files>
  <action>
Run every operation below sequentially. After each operation, check console for new JS errors.
Record pass/fail for each. If an operation fails, continue to the next (don't stop).

**2a. Table view — column sorting**
- Start in table view (default)
- Click Title header → verify rows reorder
- Click Score header → verify descending order (highest first)
- Click Source header → verify alphabetical order
- Click Status header → verify grouping

**2b. Table view — filters**
- Check "Jobicy" source filter → verify only jobicy jobs shown
- Uncheck Jobicy, check "Google" → verify only google jobs shown
- Uncheck Google
- Check "New" status filter → verify only New status jobs shown
- Uncheck New, check "Interested" → verify only Interested jobs shown
- Uncheck Interested
- Set score min to 5 → verify only scores ≥5 shown
- Reset score to 0

**2c. Table view — search**
- Type "Revenue" in search box → verify only matching titles shown
- Clear search box → verify all jobs return

**2d. Table view — status change through full pipeline**
Pick one job. Change its status through every stage, refreshing and verifying after each:
- New → Interested (refresh, verify)
- Interested → Applied (refresh, verify)
- Applied → Interview (refresh, verify)
- Interview → Offer (refresh, verify)
- Offer → Rejected (refresh, verify)
- Rejected → New (refresh, verify — back to start)

**2e. Table view — detail sheet**
- Click 3 different job titles → verify each opens the detail sheet
- For each: verify title, score, source badge are present
- For jobs with cover letters: verify cover letter content loads
- Close each sheet before opening the next

**2f. Kanban view — rendering**
- Switch to kanban view
- Verify all 6 column headers visible (may need to scroll right): New, Interested, Applied, Interview, Offer, Rejected
- Verify job cards are in the correct columns based on their current status
- Count total cards visible — should match Active count

**2g. Kanban view — switch back to table and back**
- Switch to table view → verify table renders
- Switch to kanban view → verify kanban renders
- Switch to table view → verify no rendering artifacts

**2h. Dismiss and restore cycle**
- In table view, dismiss a job via × button
- Verify Active count decreases by 1
- Verify Dismissed count increases by 1
- Click Dismissed tab → verify the job appears there
- Restore the job (change status back to New via dropdown on dismissed tab, or use × undismiss)
- Click Active tab → verify the job is back
- Refresh page → verify persistence

**2i. Final console error check**
- Read all console errors accumulated during the test
- Categorize: blocking (app crash, uncaught exception) vs cosmetic (hydration, 404)
